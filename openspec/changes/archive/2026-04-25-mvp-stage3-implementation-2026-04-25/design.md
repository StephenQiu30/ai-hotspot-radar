## Context

当前系统已经完成阶段 2：来源开启治理、标准化与原始信号入库。  
阶段 3 的瓶颈在于“同一 AI 事件跨源识别”和“事件排序稳定性”：当前实现按规范化标题精确命中分桶，标题措辞差异会导致重复事件，`X` 轻噪声也可能和单条高权重来源竞争榜位。

技术约束：
- 不变更现有 API 契约与数据库接口（MVP 稳定期）。
- 不新增采集源，避免扩大范围风险。
- 代码修改必须可回溯到 PRD P0 条目与 `acceptance` 阶段验收。

## Goals / Non-Goals

**Goals:**
- 在现有领域层能力内，提高 `cluster_raw_content` 对“标题轻差异”事件的归并能力。
- 调整评分公式与排序策略，强化多源证据优先、抑制 `X` 单源噪声进入前列。
- 用测试覆盖新增行为，保持当前接口兼容。

**Non-Goals:**
- 不引入向量/语义 embedding 或外部 NLP 服务。
- 不新增事件管理 API，不改写现有 `contracts/openapi/openapi.yaml`。
- 不引入新数据持久化表，继续沿用内存/现有模型进行 MVP 验证。

## Decisions

1. 继续把聚类逻辑放在 `backend/core/domain/hotspot_rules.py`。  
   - 原因：聚类/评分属于领域能力，保持应用层与基础设施解耦，便于后续替换。
   - 备选：放入服务层。  
   - 取舍：服务层会混杂规则与流程；领域层更易复用测试。

2. 使用轻量“文本标准化 + 词集合相似度”作为近似聚类规则。  
   - 原因：MVP 阶段追求确定性、低复杂度，不依赖外部模型。
   - 备选：逐字符编辑距离或 embedding。  
   - 取舍：编辑距离对长短标题更慢；embedding 依赖外部服务且成本高。

3. 评分公式新增“跨源信心分”后再排序。  
   - 原因：确保既保留热点传播速度（X 快速性），也要求证据面扩展。  
   - 备选：纯来源权重求和。  
   - 取舍：纯权重会放大 `X` 或单源波动；增加交叉证据门槛更符合 MVP 降噪目标。

## Risks / Trade-offs

- [Risk] 近似聚类阈值设置过严导致漏聚合。  
  - Mitigation：先偏保守（需要较高相似度），并以测试和固定样例逐步校准。
- [Risk] X 降噪规则过严导致真实快速传播事件初期下滑。  
  - Mitigation：保留 X 单源事件入池能力，仅限制其评分上限和排名权重，不做硬过滤。

## Migration Plan

- 变更为纯读路径逻辑，直接替换/增强现有聚类与评分函数。
- 验证通过后，维持原有 API 与任务编排，不需要数据库或调度变更。
- 如需回滚：恢复 `cluster_raw_content` 与 `score_hotspot_event` 为原始逻辑，任务链路可不变。

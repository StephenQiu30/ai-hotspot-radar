## Context

当前仓库已完成第一阶段文档与接口基线，并具备 `backend/core`、`services/api`、`services/worker` 的骨架实现。  
阶段2验收要求（`RawContentItem` 标准化与来源追溯）需要把“配置源是否启用”这层治理直接嵌入采集入口，而不是在路由/测试外部手工兜底。

## Goals / Non-Goals

**Goals:**
- 把标准化入口改为基于启用来源集合的执行入口。
- 为每次标准化采集保留可验证的信号存储（内存仓库），支持标准化条目的来源追溯。
- 保证单源失败只导致该源降级，不阻塞整体事件发现链路。
- 使用最少改动完成阶段2验收：不扩展 API 合同，不引入新字段。

**Non-Goals:**
- 不新增采集适配器，不新增数据库/消息队列持久化。
- 不在本阶段引入复杂的任务调度器或实时抓取策略。
- 不调整现有 API 响应字段或契约。

## Decisions

- 决策：采用 `SourceConfig.enabled` 在服务层过滤后执行标准化。  
  - 备选：在数据源构建阶段过滤。  
  - 取舍：构建阶段过滤依赖调用方约束；服务层过滤可保证任何调用路径都受治理规则统一约束。
- 决策：为标准化条目新增 `RawContentItemRepository`（内存实现）。  
  - 备选：仅返回标准化列表不落库。  
  - 取舍：阶段2验收要求“可追溯”，内存仓库可在不引入 DB 的前提下先满足验证路径。
- 决策：采集循环按来源层面 try/except，失败源仅降级记日志级别（本阶段内存测试中不持久记录失败详情）。  
  - 备选：整体抛异常中断。  
  - 取舍：与“单源失败不中断”需求一致，且避免单点抖动影响日报闭环。

## Risks / Trade-offs

- [治理边界未持久化] → 风险：来源配置变更仅在内存启动时生效；缓解：本阶段输出明确标注不含持久化扩展边界，后续再接数据库仓储。
- [原始失败未观测化] → 风险：本阶段只保证降级，不直接返回失败事件；缓解：通过单测锁定行为，下一阶段补充失败原因记录。
- [内存仓库覆盖] → 风险：每次重启或测试会清空采集记录；缓解：阶段目标为闭环验证，状态可复现。

## Migration Plan

1. 在 `HotspotDiscoveryService` 中新增“运行时来源过滤 + 采集容错”方法，替代分散的手工归并逻辑。
2. 新增 `RawContentItemRepository` 协议与内存实现。
3. 将 `services/api/dependencies.py` 与 `services/worker/bootstrap.py` 的装配入口切到新采集方法，保证 API 与 worker 共用同一入口。
4. 补充单测覆盖禁用源过滤、失败源隔离、原始信号持久化。
5. 跑 `openspec status/validate`、单元测试后归档。
